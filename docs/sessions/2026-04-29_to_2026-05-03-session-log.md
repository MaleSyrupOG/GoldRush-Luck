## 1. From idea to spec (2026-04-29 morning)

The session opened with the GoldRush platform vision: three Discord bots (Luck, Deposit/Withdraw, Poker) settling in WoW Gold against a single PostgreSQL database. The Luck bot was brainstormed and specced first.

### 1.1. Luck design — locked at v1.1

Through a sequence of clarifying questions and design sections (A through G), the Luck bot was decomposed and locked into a complete spec at `docs/superpowers/specs/2026-04-29-goldrush-luck-v1-design.md` covering architecture, database, Provably Fair, security, Discord layer, deployment, testing and documentation. The spec captured 23 locked decisions including:

- Eleven games initially → reduced to nine after collaborator's first response (Flower Poker removed).
- Uniform 5 % house edge across every game, with 1 % rake routed to a monthly raffle pool funded entirely by the rake.
- Provably Fair via per-user HMAC-SHA512 with on-demand seed rotation and a public Node + Python verifier.
- Channel-bound games, role-based command visibility (three layers), HMAC-signed Discord button `custom_id`s with TTL.
- Currency `G` (WoW Gold) stored as `BIGINT` with no `k`/`m`/`b` suffixes in input.
- Min/max bet 100 G / 500,000 G, admin-tunable via `luck.game_config`.
- Three-schema Postgres (`core`, `fairness`, `luck`) with role-isolated grants and a `SECURITY DEFINER` economic boundary so the bot's role cannot mint balance even with arbitrary SQL.

The Luck spec was followed by an implementation plan at `docs/superpowers/specs/2026-04-29-goldrush-luck-v1-implementation-plan.md` with 15 epics and ≈ 82 stories.

### 1.2. Round 2 collaborator pivot

This invalidates parts of the Luck spec but leaves the architecture intact. The Luck plan was paused. A second round of clarifying questions was prepared for the collaborator and the Luck spec is currently waiting on those answers (memory: `project_goldrush_luck_paused.md`).

### 2.2. D/W spec — locked at v1.0

Six design sections (A through F) → spec at `docs/superpowers/specs/2026-04-29-goldrush-dw-v1-design.md` (1,043 lines, 23 locked decisions in §10).

The architecture is symmetric to Luck:

- New `dw` Postgres schema alongside the shared `core` schema.
- Per-bot DB role (`goldrush_dw`) with the unique privilege of writing to `core.users` and `core.balances`.

### 8.5. Story 3.4 — `/admin setup` channel factory (commit pending push)

The fourth and final story in Epic 3 isolates the channel-creation logic so the eventual `/admin setup` command can be exercised in tests without a real Discord guild.

`goldrush_deposit_withdraw/setup/channel_factory.py` exports:

- `CategorySpec` / `ChannelSpec` — frozen dataclasses describing the canonical layout (2 categories: `Banking`, `Cashier`; 8 channels: `how-to-deposit`, `how-to-withdraw`, `deposit`, `withdraw`, `online-cashiers`, `cashier-alerts`, `cashier-onboarding`, `disputes`).
- `CategoryOutcome` / `ChannelOutcome` — per-entity result with `created` / `reused` flags and the resolved Discord id.
- `SetupReport` — aggregate result with `created_count` and `reused_count` properties for the preview embed.
- `setup_or_reuse_channels(guild, *, cashier_role_id, admin_role_id, dry_run=False, persist=None) -> SetupReport` — the entry point.

**Idempotency**. The function matches existing entities by `name + parent_category.name`. A second invocation on a fully provisioned guild creates nothing; a partial state creates only the missing pieces. The `dry_run=True` flag short-circuits every Discord-side mutation while still producing an accurate preview report — IDs are `None` for would-be-created entities, real for already-existing ones.

**Permission matrix**. Hard-coded against spec §5.3, with one explicit substitution: discord.py 2.4.0 does not expose `view_private_threads` (Discord folded the read side of that bit into `manage_threads`), so the cashier role on `#deposit` and `#withdraw` is granted `manage_threads` instead. Admin gets full view + send everywhere; @everyone is denied view on cashier-only channels and on `#disputes`; the bot gets `manage_messages` on the read-only welcome channels and `manage_channels` + `manage_threads` on `#deposit` because the per-ticket private channels are created off it.

**Persistence boundary**. The module is intentionally DB-agnostic. The `persist` parameter accepts an async callback that takes `{channel_key: discord_id}`; when `/admin setup` lands in Epic 10, it will pass a callback that writes the map into `dw.global_config`. The callback is skipped automatically on `dry_run`.

**Reality alignment note**. The live server already runs with renamed channels (`#cashier-requests` instead of `#cashier-alerts`, plus `#audit-log` and `#admin-commands` for admin tooling). ## 12. Epic 8 — Background workers

### 12.1. Why this epic next

After Epic 9 closed the disputes + blacklist surface, Epic 8 is the
operational hardening layer. Without these workers, a ticket left
open by a user who walked away or a cashier who got distracted
stays "open" forever; balances are locked indefinitely on
abandoned withdraws; idle cashiers count as online in the live
roster. Each worker plugs one of those holes.

Story 8.4 (online cashiers embed updater) was already shipped as
part of Story 4.5 — confirmed via the `online_cashiers_updater_started`
log line in production. Five remaining stories landed in five
atomic commits.

### 12.2. Scaffolding — `PeriodicWorker` base class

The five new workers share the same shape: an asyncio loop that
runs `tick()` every N seconds, with idempotent `start()`, awaitable
`stop()`, broad-except in the loop, and an immediate first tick on
start. The pre-existing `OnlineCashiersUpdater` in
`goldrush_deposit_withdraw/cashiers/live_updater.py` already
implements this pattern in ~50 lines; rather than duplicate it
five times, Story 8.1 introduces
`goldrush_deposit_withdraw/workers/_periodic.PeriodicWorker` with
the scaffolding factored out. Subclasses override
`async def tick(self) -> None`. `OnlineCashiersUpdater` is left as
is to avoid touching working code; a future cleanup story can
migrate it.

4 base-class tests cover: initial tick happens immediately,
`start()` is idempotent (no double-rate), `stop()` waits for the
loop to exit, the loop survives an exception in `tick`.

### 12.3. Story 8.1 — `ticket_timeout_worker`

Every 60 s, scan both `dw.deposit_tickets` and `dw.withdraw_tickets`
for `status IN ('open','claimed') AND expires_at < NOW()`. Cancel
each via the canonical `cancel_deposit` / `cancel_withdraw` SDFs
(refunds locked balance for withdraws). System actor (id=0) and a
reason like `auto-cancel: expired (open|claimed ticket ...)` keep
the audit row self-explanatory.

Claimed-side cancellations also surface in `#audit-log` via
`audit_ticket_cancelled` (System actor) — admins notice cashiers
who let claims go stale until the timeout fires. Open-side
cancellations are routine and stay in the SDF audit row only.

Idempotency: `ticket_already_terminal` is swallowed; a crash
mid-loop or a race with admin force-cancel converges to the same
end state.

6 worker tests + the 4 base-class tests = 10 new passes. Suite
**380 → 390**.

### 12.4. Story 8.2 — `claim_idle_worker`

Every 60 s, two distinct deadlines (each its own SQL UNION across
deposit + withdraw):

- `last_activity_at < NOW() - 30 min` → `release_ticket` (system
  actor) AND repost cashier alert in `#cashier-alerts` so the next
  cashier picks the ticket up.
- `claimed_at < NOW() - 2 h` → `cancel_deposit` / `cancel_withdraw`.

`tick` returns a `TickSummary(released, cancelled)` so observability
stories (Story 11.x) can emit per-branch metrics without
re-instrumenting.

8 tests covering the two branches, the idempotent races
(`ticket_not_claimed`, `ticket_already_terminal`), the mixed pass,
and the empty no-op. Suite **390 → 398**.

### 12.5. Story 8.3 — `cashier_idle_worker`

Every 5 min. Scans `dw.cashier_status` for rows in
`status='online' AND last_active_at < NOW() - 1 h` and auto-offlines
each via the new `dw.expire_cashier(p_discord_id)` SDF (migration
`20260503_0016_dw_expire_cashier`).

The session row gets `end_reason='expired'` (distinct from
`'manual_offline'` written by `dw.set_cashier_status`); the audit
row reads `cashier_status_offline_expired`. Splitting the verb
keeps reporting on "manual offlines vs. idle timeouts" a single
GROUP BY.

New typed exception `CashierNotOnline` on the
`cashier_not_online` sentinel; the worker swallows it on the
idempotent race where a manual `/cashier-offline` beats the worker.

3 worker tests + 2 wrapper tests = 5 new passes. Suite
**398 → 403**.

### 12.6. Story 8.5 — `stats_aggregator`

Every 15 min. Walks `dw.cashier_stats` and recomputes two derived
columns per cashier in ONE UPDATE statement:

- `avg_claim_to_confirm_s` — moving average over the last 100
  confirmations across BOTH ticket families (subquery LIMIT 100).
- `total_online_seconds` — SUM of `cashier_sessions.duration_s`.

Plain SQL — no SDF needed because the bot's role already has the
required SELECT on tickets/sessions and UPDATE on `cashier_stats`.
Idempotent recompute from scratch (no "since last run" pointer)
keeps the worker simple at our current scale.

3 tests: empty no-op, one UPDATE per cashier, UPDATE statement
covers both fields and both ticket families. Suite **403 → 406**.

### 12.7. Story 8.6 — `audit_chain_verifier`

Every 6 h, walks `core.audit_log` recomputing the HMAC chain.

Architectural choice: the bot's `goldrush_dw` role does NOT have
SELECT on `core.audit_log` (deliberate — read access stays gated to
`goldrush_readonly`). So the verifier itself runs as a SECURITY
DEFINER fn, `core.verify_audit_chain(p_from_id, p_max_rows)`
(migration `20260503_0017_core_audit_chain_verifier`), which walks
the rows with admin privileges and returns a thin
`(checked_count, last_verified_id, broken_at_id)` table. The bot's
role gets EXECUTE on the verifier but never gets SELECT on the
underlying table.

The canonical jsonb composition in the verifier MUST match
bit-for-bit what `core.audit_log_insert_with_chain` (migration
0002) writes — both sides use `jsonb_build_object(...)` over the
same column set so the bytes line up.

Worker behaviour:
- Reads `dw.global_config.last_verified_audit_row_id` as the
  resume pointer (defaults to 0 on first run).
- Calls the SDF with that id and a 1000-row cap.
- On HEALTHY: INFO `audit_chain_verified` + UPSERT new last id.
- On EMPTY range (no new rows): skip the UPSERT.
- On BROKEN: CRITICAL `audit_chain_break` log line with
  `broken_at_id`; does NOT advance the pointer so the next iteration
  re-checks the same range after admin remediation.

`/admin-verify-audit` slash command runs the same `tick()` and
reports inline (✅ healthy / 🚨 break). Story 11.3 (Alertmanager)
will route the CRITICAL log to the on-call channel; today operators
find it via Loki.

5 worker tests + 1 admin cog test = 6 new passes. Suite
**406 → 412**.

## 14. Post-Epic-10 hot fixes & cashier onboarding

After deploying Epic 10, smoke-testing the live bot surfaced one
real bug and one missing-but-obvious feature. Both landed as small

### 14.1. Bug — `/admin-force-close-thread` race with archive

Logs caught three identical stack traces inside the first hour of
post-Epic-10 testing:

```
discord.errors.Forbidden: 403 Forbidden (error code: 50083):
  Thread is archived
  File "/app/goldrush_deposit_withdraw/cogs/admin.py", line 460,
  in force_close_thread
    await interaction.response.send_message(...)
```

Reproduction: an admin runs `/admin-force-close-thread` from inside
the very thread they're archiving. The handler called
`thread.edit(archived=True)` FIRST, then
`interaction.response.send_message(...)` — but archiving the thread
cuts off the response channel, so the ack 403'd. The archive itself
worked; the user just saw "interacción fallida".

**Fix**: ack the interaction with a `⏳ Archiving...` ephemeral
BEFORE the archive call (while the thread is still alive). After
the archive lands, edit_original_response promotes the ephemeral to
✅ / ❌. The audit-log post sits between the archive and the edit
because it goes to `#audit-log` (never the now-archived thread) so
it's the last source-of-truth side effect.

3 new regression unit tests using dependency-injected fakes assert
the call order. Suite 443 → 446.

## 15. Epic 11 — Observability

### 15.1. Why this epic next

After Epic 10 closed the admin operational surface, observability
is the bridge between "the bot works" and "we can prove the bot
works in production". Spec §7.3 asks for 10 Prometheus metrics, a
Grafana dashboard, and 5 Alertmanager rules — landing those means
Aleix can leave the bot running and trust that anything anomalous
surfaces as a Discord alert before users notice.

Three stories, three atomic commits.

### 15.2. Story 11.1 — Prometheus metrics + DB refresher worker

**Type choice.** Spec §7.3 names the cumulative-count metrics with
the `_total` suffix, which Prometheus convention reserves for true
Counters. But the bot restarts on deploy would reset a Counter to
zero — the DB still holds the cumulative state. So I went with
Gauges of absolute totals read from the DB on each refresh.
Prometheus `deriv()` / `increase()` work on Gauges for rate panels;
the few cases where Counter semantics matter are handled by
`deriv()` on the Gauge.

Histograms stay Histograms — populated via `observe()` at confirm
time in the ticket cog. Two hooks added in `cogs/ticket.py`:

- After `confirm_ticket_dispatch`, observe the cog-side latency
  via `time.perf_counter()` deltas.
- After Success branch, observe the claim→confirm gap via
  `datetime.now(UTC) - ticket_row["claimed_at"]`.

**Architecture.** Custom `CollectorRegistry` (not the global) so
the test suite scrapes a known set without picking up Python
process / GC defaults. `start_metrics_server(port=9101)` from
`DwBot.on_ready`. `MetricsRefresherWorker` (subclasses Story 8.1's
`PeriodicWorker`) runs every 30 s and `refresh_from_db(pool)` does
the per-metric SELECTs with try/except per metric so one bad query
doesn't poison the rest.

**Compose footprint.** Bot keeps `goldrush_net` only — the
canonical compose.yml does NOT touch foreign networks. A new
opt-in overlay (`compose.observability.yml`) documents two routing
patterns: publish on host loopback (default), or attach to a
foreign observability docker network (operator edits to set the
network name). The operator picks. **No VPS changes happen
automatically.**

7 unit tests, suite 448 → 455.

### 15.3. Story 11.2 — Grafana dashboard JSON
