## 1. From idea to spec (2026-04-29 morning)

The session opened with the GoldRush platform vision: three Discord bots (Luck, Deposit/Withdraw, Poker) settling in WoW Gold against a single PostgreSQL database. The Luck bot was brainstormed and specced first.

### 1.1. Luck design — locked at v1.1

Through a sequence of clarifying questions and design sections (A through G), the Luck bot was decomposed and locked into a complete spec at `docs/superpowers/specs/2026-04-29-goldrush-luck-v1-design.md` covering architecture, database, Provably Fair, security, Discord layer, deployment, testing and documentation. The spec captured 23 locked decisions including:

- Eleven games initially → reduced to nine after collaborator's first response (Flower Poker removed).
- Uniform 5 % house edge across every game, with 1 % rake routed to a monthly raffle pool funded entirely by the rake.
- Provably Fair via per-user HMAC-SHA512 with on-demand seed rotation and a public Node + Python verifier.
- Channel-bound games, role-based command visibility (three layers), HMAC-signed Discord button `custom_id`s with TTL.
- Currency `G` (WoW Gold) stored as `BIGINT` with no `k`/`m`/`b` suffixes in input.
- Min/max bet 100 G / 500,000 G, admin-tunable via `luck.game_config`.
- Three-schema Postgres (`core`, `fairness`, `luck`) with role-isolated grants and a `SECURITY DEFINER` economic boundary so the bot's role cannot mint balance even with arbitrary SQL.

The Luck spec was followed by an implementation plan at `docs/superpowers/specs/2026-04-29-goldrush-luck-v1-implementation-plan.md` with 15 epics and ≈ 82 stories.

### 1.2. Round 2 collaborator pivot

The collaborator returned with a different vision: 5 games (Blackjack, Towers, Mines, Hi-Lo, Keno), a persistent button-driven UX flow, and new systems (rank, rakeback). This invalidates parts of the Luck spec but leaves the architecture intact. The Luck plan was paused. A second round of clarifying questions was prepared for the collaborator and the Luck spec is currently waiting on those answers (memory: `project_goldrush_luck_paused.md`).

### 2.2. D/W spec — locked at v1.0

Six design sections (A through F) → spec at `docs/superpowers/specs/2026-04-29-goldrush-dw-v1-design.md` (1,043 lines, 23 locked decisions in §10).

The architecture is symmetric to Luck:

- New `dw` Postgres schema alongside the shared `core` schema.
- Per-bot DB role (`goldrush_dw`) with the unique privilege of writing to `core.users` and `core.balances`.
- Twelve `dw.*` and two `core.*` `SECURITY DEFINER` functions form the economic boundary: the bot can only mint or destroy balance through them, and they encode every invariant.
- Treasury accounting via a single row at `core.balances[discord_id=0]`.
- `/admin setup` command will auto-create the canonical channel structure (Banking + Cashier categories with eight channels) on first install.

### 5.2. Tooling fixes captured along the way

- `psycopg2-binary 2.9.10` added to runtime dependencies. SQLAlchemy defaults to psycopg2 for plain `postgresql://` URLs and Alembic's env.py uses sync engine_from_config; without a sync driver migrations cannot run.
- Dockerfile.dw rewritten to build the venv at `/app/.venv` from the start so script shebangs (`alembic`, `dotenv`, `mako-render`) reference the correct interpreter location after the runtime-stage copy. Previously the shebang pointed at `/build/.venv/bin/python` which only existed in the discarded builder stage.
- `audit_log_insert_with_chain()` updated to call `public.hmac(...)` fully qualified, because the SECURITY DEFINER `search_path = core, pg_catalog` excludes the public schema where pgcrypto installs its functions, with `'sha256'::text` cast to disambiguate the overload.

### 5.3. Smoke tests (locally) and VPS apply

A fresh local Postgres ran all 12 migrations cleanly. End-to-end smoke tests verified:

- Deposit cycle: register cashier char → set online → create ticket → claim → confirm. User balance went 0 → 50,000 G. Audit log produced 5 rows with correct hash chaining (`prev_hash` of row N equals `row_hash` of row N-1; the first row's `prev_hash` is NULL).
- Withdraw cycle: create_withdraw_ticket(30,000) → claim → confirm. User balance went 50,000 → 20,000 (full debit), treasury went 0 → 600 (the 2 % fee), `amount_delivered` persisted as 29,400 on the ticket.
- Permission boundary: connecting as `goldrush_luck` and attempting `UPDATE core.balances` and `INSERT core.users` both returned `permission denied`. Connecting as `goldrush_admin` and attempting `UPDATE` and `DELETE` on `core.audit_log` both raised the trigger exception `audit_log is append-only`.

The same migrations were then applied to the live VPS Postgres (`ALTER DATABASE goldrush SET app.audit_chain_key = '<hex>'` first), and the bot container was rebuilt and restarted with the new image. Final VPS state at session end:
